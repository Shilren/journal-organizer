# 项目总结：从想法收集器到个人知识 Agent

## 1. 项目要解决的问题

人的许多重要认知不是在“正式写作”时出现，而是在走路、通勤、面试结束或情绪发生变化的几分钟里出现。传统笔记工具的问题在于记录成本高；普通 AI 总结的问题在于，它会把个人思维流改写成标准答案，长期看反而丢失了人的原始判断。

Journal Organizer 的目标是建立一条低摩擦闭环：

```text
随手表达 -> 忠实整理 -> 自动归档 -> 周期复盘 -> 个性化检索 -> 再次思考
```

更深一层，它希望把“保存自我”和“延续自我”变成两种可以协作的产品能力：Journal Bot 保存未经重写的一手经验，Second Me 让这些经验在新的处境里继续参与判断。完整价值与产品理念见 [PHILOSOPHY.md](PHILOSOPHY.md)，真实个人使用方式见 [PERSONAL-CASE-STUDY.md](PERSONAL-CASE-STUDY.md)。

## 2. 系统如何演进

### 阶段一：忠实记录

最早的核心只有一条原则：整理语言，不重排思路。AI 去除口头禅、修复转写和标点，但保留第一人称、原始顺序和当时的迟疑与转折。确定性的文件写入交给 Python，而不是每次让模型自由生成 Markdown。

### 阶段二：移动端和可靠性

飞书 Bot 让手机成为低摩擦入口。真实使用很快暴露出更重要的需求：电脑睡眠、长连接断开、模型调用超时、同一事件重复消费。系统因此加入历史补抓、成功后标记、消息 ID 去重、有限重试和 launchd 自愈。

### 阶段三：周期复盘

零散记录需要形成时间维度上的反馈。周复盘把一周的一手材料整理成连续文章，关注主线、变化和矛盾。为了避免把 AI 生成内容再次当作一手事实，复盘文件被明确标记为二级材料。

### 阶段四：个人知识问答

单次关键词搜索无法回答“我真正重视什么”“最近发生了什么变化”这类问题。Knowledge Bot 因此加入分类路由、片段级检索、时间权重、材料可信度、长期自我模型和短期对话记忆。

### 阶段五：职责分离

最终系统拆成两个 Bot：Journal Bot 追求忠实和可靠；Second Me 追求理解和推断。读写权限分离后，Prompt 更简单，故障边界也更清楚。

这不仅是工程拆分，也是产品立场：记录时，AI 不应抢走用户的叙事权；回答时，AI 又需要超越复述，真正把长期材料内化成一个连续视角。

## 3. 知识层级

系统不把所有 Markdown 看成同一种材料：

| 层级 | 内容 | 默认权重 | 用途 |
|---|---|---:|---|
| 一手材料 | 用户原始记录与忠实清洗正文 | 高 | 具体经历、价值判断、真实措辞 |
| 二级材料 | AI 生成的周复盘 | 低 | 成长趋势、跨日主线、阶段变化 |
| 长期模型 | 用户主动维护的身份、目标、原则、风格 | 高且稳定 | 回答立场和人格一致性 |
| 短期上下文 | 最近几轮聊天 | 局部优先 | 指代、连续追问和当前意图 |

当材料冲突时，具体问题优先回到一手记录；当问题询问变化过程时，二级复盘得到额外权重。

## 4. 关键工程决策

### 模型负责语义，脚本负责确定性

分类、清洗和写作需要语言模型；日期、追加、去重、撤回和文件路径必须由代码控制。这减少了覆盖旧内容、格式漂移和重复归档。

### 先检索，再生成

Knowledge Bot 先在本地缩小材料范围，再调用 Codex。这样节省上下文，也让分类优先、时间权重和材料可信度可以被明确控制。

### 原始记录保留在折叠区

清洗后的正文适合阅读，原始文本适合审计。两者同时保存，用户可以随时检查 AI 是否改变了含义。

### 双应用隔离事件流

飞书长连接消费者会竞争同一应用的事件。因此两个 Agent 必须使用两个 App/profile。这不是人格设定问题，而是消息传输层的所有权问题。

### append-only 周复盘索引

每次成功归档时额外写一条 JSONL 事件；撤回则写 tombstone。周复盘优先读取这份索引，Vault 扫描只做回退，降低 macOS 隐私权限导致定时任务读不到 Obsidian 的概率。

## 5. 隐私与公开策略

GitHub 仓库公开系统，不公开用户：

- 代码、Prompt 模板、示例配置和匿名样例可以公开。
- App Secret、profile 凭证、聊天 ID、消息 ID、日志和本机路径不可公开。
- 私人知识库正文与真实 `self_model.md` 永远只留在本机。
- Knowledge Bot 使用只读 Vault 和 Codex `read-only` sandbox。

## 6. 当前限制

- 默认检索是轻量词法匹配，适合个人规模 Vault；数万条以上建议切换 SQLite FTS 或本地向量索引。
- macOS 后台服务已经覆盖，Linux systemd 尚未提供。
- 飞书语音消息仍依赖平台转写成文字；原始音频转写不在当前项目范围内。
- 长期自我模型需要用户主动维护，系统不会把每次情绪自动升级为永久价值观。

## 7. 下一步路线

- 增加 SQLite FTS5 与可选 embedding 检索。
- 为自我模型提供“候选更新 -> 人工确认 -> 生效”的演进工作流。
- 增加知识冲突检测和观点时间线。
- 提供 systemd、Docker 和其他消息入口。
- 增加本地评测集，衡量忠实度、召回率、人格一致性和短期上下文承接。
